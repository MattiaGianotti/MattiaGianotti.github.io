The Magic Room

The Magic Room is an innovative technological solution that transforms any room into a “smart” space where lights, immersive projections on walls and floors, music, sounds, aromas, and physical materials are digitally controllable, programmable, and interactive. The result is a “magic” space where children are exposed to multisensory stimuli and can interact with physical materials and multimedia content through movements, gestures, and object manipulation. Magika, its operating system, enables playful and educational activities (individual or collective) that engage all the senses. Teachers can use a tablet to control and configure each activity, personalizing the experience to meet the specific needs of the children.

The Magic Room is powered by a sophisticated Internet of Things-driven solution integrated with Augmented Reality. It features a multilayered, multimodular architecture that ensures maintainability, scalability, and the potential for future upgrades and project evolution.

Timeframe of the project: 2018 - present

HOW DOES IT WORK?
Smartify the space!

To create a Magic Room, we start with any empty room available, requiring at least 3x3 meters of floor space and a white wall.

We then conceal as much as possible of the technological components from children's view by embedding them in the ceiling, within the walls, or inside everyday objects modified ad hoc to function as controllers of the space. Most of our appliances are off-the-shelf devices, which we integrate using API calls over the local network to enhance system stability and manageability. When no suitable commercial product was available, we developed custom Smart Objects.

The final result is a human-centered approach to the Internet of Things (IoT), employing a centralized control system to coordinate various stimuli in response to user actions. For motion tracking, we use cameras and wall-touch identification powered by LiDAR technology. Artificial intelligence models are employed to analyze the data and determine user interactions.

The technology inside can vary from installation to installation, but the main components are:
  1. Two projectors: one projecting on the front wall (A) and one projecting on the floor (B).
  2. Audio Dolby Surround 5.1 (C).
  3. Smart Material (D): material samples identifiable by the system through Smart Objects (E).
  4. Bubble emitter (F).
  5. Aroma emitter (G).
  6. Smart lighting system controllable in intensity and color, either portable (H) or fixed on the ceiling (I).
  7. Motion tracking device (J), now powered by a custom solution using Google MediaPipe.
  8. Control tablet (K).
  9. Text-to-Speech system (L).
  10. LiDAR sensors (M).
What was my role?

My initial role was as the lead developer. I defined, designed, and implemented the global architecture of the system, covering both its functional and active components.

The goal was to create a solution with expressive power for human-driven IoT (Internet of Things) powered smart spaces, using an approach similar to what ROS (Robot Operating System) represents for robotic design.

After the initial release of the first six activities, my role evolved into that of a Technical Project Manager. In this role, I supervised a team of over 20 people who developed new activities, improved input and output mechanisms, and revised control structures to enhance the system's usability and stability. I also continued to develop and improve the activities, integrating them into the production system to make them available to care centers and schools where the Magic Room is installed (currently five across Italy). Additionally, I became the designer for the new installations.

Over the years, I managed a team of more than 20 people from diverse backgrounds, including computer scientists, psychologists, linguists, physiatrists, teachers, caregivers, and designers, just to name a few. I learned to value the unique contributions each individual brought to the project and how to organize and delegate tasks effectively to optimize the development process, significantly reducing the time required to create a single experience.

HOW DID WE IMPLEMENTED THEM?

To standardize the development process and allow students and researchers to create new experiences in the shortest time possible, we have created a set of support tools.

The most important element is the starting Magic Room UnityPackage.
The UnityPackage I developed is a startup system for anyone who needs to create a new activity for the Magic Room. Among its several functions, it:
  • Offers the primitives to control the different appliances distributed in the Magic Room.
  • Hides the details of the connections to the system.
  • Provides synchronization primitives for seamless integration with the global system.
  • Includes a setup scene and a generative game configuration menu creation.
  • Automatically generates configuration details, a control panel, and a default standard package for smooth integration into the global system.
The package offers both traditional C# scripts for development, unity prefab elements and Unity editor scripts to wrap different models, providing a clearer set of functionalities, documentation, and testing capabilities (available only in the Magic Room), while masking the more complex functionalities of the system.

About the orchestration:

Our experience orchestrator is a generative system designed to manage multiple processes. One of its main functionalities is to provide the configuration to run personalized experience both to the users on the Magic Room, and to caregivers by customizing the tablet visualization ad hoc for the situation. A second key function is to activate and maintain the different middleware programs that enable the Magic Room to function properly. A third key function is to determine which activities should be visible to users when they are ready, avoiding disruptions with loading screens and offering the illusion of a seamless, magical experience for the users.

About the remote control system:

Our system must be controlled by caregivers while staying with the children "in the field," thus not being able to use the mouse and keyboard of the controlling PC. For this reason, we have designed a configurable tablet interface that allows any user to control the system. In the tablet view, it is possible to:

  • Select one of the available activities
  • Configure the activity to play
  • Package up the activities to play in the session
  • Select the currently playing child among the available players
  • Use general-purpose commands available in any activity
  • Simulate the behavior of the system's sensors in case of sensor errors
  • Terminate the experience or enter emergency mode if the user is experiencing difficulties
